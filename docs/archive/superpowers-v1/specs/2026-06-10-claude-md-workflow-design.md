# Spec: `my-Claude-Code` 项目级 CLAUDE.md 工作流设计

- **日期**：2026-06-10
- **作者**：主线程（与项目负责人协作产出）
- **状态**：已通过设计复审，待用户复审

## 1. 摘要

为 `my-Claude-Code` 项目落地一份**项目级 CLAUDE.md**，约束主线程与子线程的角色分工、状态外化策略、Superpowers 技能选用规则，使主线程专注于"对齐需求、把握方向"，具体工作交由子 Agent / 空上下文子线程承担，且整套约定可由未来的主线程低摩擦接手。

## 2. 背景与动机

项目 `my-Claude-Code` 是关于 Claude Code 自身学习与实践的项目。仓库里已有 `references/s01-s20/` 作为参考材料。日常使用中：

- 主线程（与人对话的线程）容易被大量上下文撑爆
- 大量开发/测试/验收工作在主线程里直接做，挤占需求对齐的带宽
- 没有项目级契约约束主线程"该做什么、不该做什么"
- 没有约定把项目状态外化到文件，未来主线程接手要重头捋

## 3. 目标

- 落地一份 `CLAUDE.md`，明确**主线程边界**、**子线程派活/收工契约**、**状态外化约定**、**Superpowers 技能选用规则**
- 主线程寿命延长：上下文可以外化给文件，未来主线程从文件继续
- 子线程可以**空上下文**启动，**指向性 prompt** 就能开工
- 任务可拆时**优先并发**派活，不串行阻塞主线程

## 4. 非目标

- **不**预置 `docs/superpowers/state/` 下的状态文件（采用「极简骨架」方案，按需创建）
- **不**预置派活/收工模板文件
- **不**重写 `references/s01-s20/`
- **不**约束 `references/` 的内容（它是用户提供的启发性参考）
- **不**给主线程任何写文件/编辑文件的权限

## 5. 关键设计决策

| 维度 | 决策 |
|------|------|
| 主线程边界 | 查阅型：A（只读）+ B（单条试错命令） |
| 状态触发点 | 决策点 / 验收点必写 |
| 状态格式 | 纯 markdown |
| 状态位置 | `docs/superpowers/` 下 |
| 状态布局 | 主题分多文件（4 个 state 文件） |
| 子线程粒度 | 动态（看任务） |
| Superpowers 流程 | 轻量参考，不强制 |
| 派活契约 | 指向型（路径+任务描述） |
| `references/` 角色 | 启发性参考，可读可选，不进状态文件 |

## 6. 设计细节

### 6.1 项目定位与基本姿势

`my-Claude-Code` 是一个**关于 Claude Code 自身**的学习与实践项目。两类资产：

- `references/s01-s20/` — 用户提供的**启发性参考**。主/子线程可主动阅读以借鉴实现方式；若参考不是最优，按更优的来；不写进任何项目状态文件
- 其它仓库内容 — 项目自身的学习产出

基本姿势：

- 主线程以**查阅 + 派活 + 决策**为主
- 具体的开发、测试、调研、子任务实施由**子 Agent / 空上下文子线程**承担
- 上下文管理策略：决策/验收节点上**外化到文件**，未来主线程可低摩擦接手

### 6.2 主线程边界

**允许（A+B）**：

- **A. 只读查阅**：读项目文件、查 `references/s01-s20/`、查 git 历史、看进程输出、看子 Agent 返回的报告
- **B. 试错型单条命令**：跑一条命令验证想法，例如 `git status`、`python -c "1+1"`、`man xxx`

**禁止（C/D/E）**：

- **C**：编辑或新建任何文件
- **D**：实施完整功能、调试、跑测试套件
- **E**：与外部世界交互（`git push`、PR、`gh`、装包、调外部 API）

B 的边界是"单条"——若是多步骤、或者可能改文件（如 `git commit` 会改 `.git/`），必须派活。

### 6.3 子线程派活与收工契约

**派活契约（主线程 → 子线程）**：

- 派活 prompt 必须包含：**(1) 任务描述**、**(2) 相关文件路径**（spec/plan/state/decision）
- 子线程**自己**去读路径指向的文件，不在 prompt 里复制大段内容
- 任务可拆为多视角/多阶段/互相独立时，主线程应**主动调 `dispatching-parallel-agents`** 拆成并发子线程

**收工契约（子线程 → 主线程）**：

- 子线程完工时**必须**给出 4 块结构化产物：
  1. 改动清单（文件 + 简述）
  2. 关键决策/取舍（如有）
  3. 验证结果（跑了什么/看了什么）
  4. 未决问题（如有）
- 不应在主线程上下文里留大段历史——主线程只接收上述四块

**粒度（动态）**：

- 单一原子动作（如修一行） → 1 个子线程
- 一个完整小特性 → 1 个子线程自带 `writing-plans`
- 多模块/多视角 → 多个子线程并发

### 6.4 状态外化

**位置**：`docs/superpowers/` 下分三目录：

- `specs/` — 需求/设计文档（`brainstorming` 技能产出）
- `plans/` — 实施计划（`writing-plans` 技能产出）
- `state/` — 状态文件（本节核心）

**state/ 下 4 个文件**（按主题分多文件）：

- `STATE.md` — 项目当前全景：愿景、当前目标、下一动作
- `decisions.md` — 决策日志：按时间倒序，每条决策记日期、决策、原因、影响
- `progress.md` — 派活/收工流水：按时间倒序
- `open-questions.md` — 未决问题：需要用户决策才能继续的事

**何时写**：

- 决策/验收点 → 至少更新 `STATE.md` 和 `decisions.md`
- 子线程完工 → 至少更新 `progress.md`
- 任何阻断子线程的问题 → 写入 `open-questions.md` 并提示用户

**子线程读什么**：

- 接到任务时**先读** `STATE.md` 了解项目当前位置
- 涉及历史决策时**按需读** `decisions.md`
- 实现方式选择时**可主动**读 `references/s01-s20/`，但不是必读，也不是项目状态来源

**格式约束**：

- 纯 markdown
- 文件头部用 H1 + 简短说明（如 `STATE.md` 顶部放「项目全景：...（最后更新：YYYY-MM-DD）」）
- 时间统一 ISO 日期 `YYYY-MM-DD`

### 6.5 Superpowers 流程：按场景选用

| 场景 | 建议技能 |
|------|---------|
| 用户说"想做个 X" / "改个行为" / "加个功能"，含义/边界/取舍还没定 | `brainstorming` |
| brainstorming 后产出 spec | `writing-plans` 把它转成可执行 plan |
| plan 已就绪，开始动手 | `executing-plans`（独立 session）或 `subagent-driven-development`（当前 session 子 Agent 派活） |
| 接到可拆解为多个独立子任务的工作 | `dispatching-parallel-agents` |
| 跑测试 / 验收子线程产出 | `test-driven-development`（如适用）/ `verification-before-completion` |
| 收到 code review 反馈 | `receiving-code-review` |
| 遇到任何 bug / 测试失败 | `systematic-debugging`（**强制**：先定根因再修） |
| 准备合并 / 收尾 | `finishing-a-development-branch` |
| 大改前自查或大改后自查 | `requesting-code-review` |

**强制项**（无论流程严不严）：

- 遇到任何 bug 或测试失败 → **必须**先 `systematic-debugging` 再修
- 收尾（说"做完了"前） → **必须** `verification-before-completion`
- 接到看似模糊的反馈（"代码建议改改"） → **必须** `receiving-code-review` 而不是直接动手

**不强制**：

- 不强制每次都跑完整 brainstorming → spec → plan → 实施的闭环
- 小修小补可以跳过整套
- 但若用户表达"想做个 X"且含义未澄清，主线程应主动调 `brainstorming`

### 6.6 异常路径与上下文管理

**子线程失败的处置**：

- 主线程不直接在主线程修补，而是**重新派活**，prompt 里附前次失败的关键信息（读了什么、试了什么、错在哪）

**主线程上下文膨胀的处置**：

- 自我评估剩余空间不足时：
  1. 把当下共识**外化**到 `STATE.md`（和 `decisions.md` 如有）
  2. 提示用户：「对话已较长，已把当前状态写入 STATE.md，建议『新开会话』从 STATE.md 继续——请把 STATE.md 路径作为新会话的第一句指令」
- 用户主动说"我们暂停 / 收一下" → 同样外化

**冲突解决**：

- 子线程产出与 `STATE.md` / `decisions.md` 矛盾时 → 以状态文件为准，子线程在 completion report 中标记"与现状 X 矛盾"，主线程裁决
- 决策变更 → **追加**新决策到 `decisions.md`（带"覆盖 X 决策"标注），不删旧决策

## 9. 风险与权衡

- **CLAUDE.md 一旦写入，主线程每次都会读到**。所以语言/边界必须精炼，避免歧义
- **状态文件按需创建意味着首次实战时主线程要主动建 `STATE.md` 等**。这是个一次性摩擦，spec 阶段接受
- **不预置派活/收工模板意味着每次派活主线程要现想 prompt 结构**。权衡：保持仓库干净 vs 复用；本次选前者，必要时后续可补
- **`references/` 角色定为"可读可选"**——主线程/子线程是否会真去读，依赖纪律。后续可观察，必要时调严
